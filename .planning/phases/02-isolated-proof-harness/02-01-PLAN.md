<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintainer input → proof harness | Repo and manifest paths are untrusted filesystem input |
| proof harness → artifact consumers | Generated JSON evidence becomes the basis for later acceptance logic |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T | `scripts/gdscript-proof.sh` run index | mitigate | Write the run index from existing wrapper/status files and record missing wrappers explicitly so evidence cannot silently appear complete |
| T-02-02 | I | runtime state isolation | mitigate | Keep `HOME`, `XDG_CONFIG_HOME`, and `XDG_CACHE_HOME` under run-owned state roots and expose those paths in machine-readable artifacts |
| T-02-03 | R | incomplete-path reporting | mitigate | Persist repo stage status, failure messages, and CLI capture notes in both `repo-meta.json` and the run index so later phases can audit what happened |
| T-02-04 | E | external repo checkout handling | accept | D-05 keeps clone/materialization out of scope; the harness only consumes provided local paths and does not escalate privileges |
</threat_model>
