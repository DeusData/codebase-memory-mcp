## Summary

The codebase already parses GDScript with Tree-sitter and includes both `variable_statement` and `export_variable_statement` in `gdscript_var_types` (`internal/cbm/lang_specs.c`). There is also evidence that `@export var ...` currently parses in extraction tests (`tests/test_extraction.c`, existing `gdscript_defs_anchor_and_methods`).

However, variable export status currently flows through generic `cbm_is_exported(name, lang)` behavior in `push_var_def(...)` (`internal/cbm/extract_defs.c`), which does not encode GDScript-specific `@export` semantics. For GDScript specifically, this can over-mark variables as exported because generic fallback behavior is too broad.

Current GDScript linking conventions already use a synthetic script anchor (`cbm_gdscript_anchor_qn`) and parent-class wiring for methods/signals. Variable definitions can be brought into the same linking fidelity by assigning GDScript variable qualified names under the active anchor/class context and deriving `is_exported` from AST annotation/statement semantics instead of generic name heuristics.

## Architecture Patterns

### Pattern 1: Language-specific semantics in extraction layer
The codebase centralizes language-specific semantic differences in extraction (`extract_defs.c`, `extract_imports.c`) while keeping shared infrastructure generic.

### Pattern 2: Anchor-based GDScript linkage
GDScript methods/signals are linked under script/class anchor QNs. Variable linking should follow the same pattern so downstream graph queries stay coherent.

### Pattern 3: Regression-first stabilization
Existing phases repeatedly enforce behavior with native C tests plus proof-layer regressions. This phase should follow the same behavior-first test contract.

## Common Pitfalls

1. **Pitfall:** Treating `@export` as syntax-only without index semantics.
   - **Avoid by:** Asserting persisted `is_exported` behavior in pipeline-level tests.

2. **Pitfall:** Using only name-based export heuristics for GDScript variables.
   - **Avoid by:** Inspecting AST node kind / annotation ancestry for `@export`.

3. **Pitfall:** Introducing `@export` support but leaving variable QNs unlinked to script anchor.
   - **Avoid by:** Ensuring variable QN/parent linkage follows established GDScript anchor conventions.
